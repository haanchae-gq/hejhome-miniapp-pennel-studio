## 요약 판정

**부분 타당 — 단, 지금 스튜디오가 닿는 곳은 키즈노트 상품의 한 갈래뿐이다.**

| 키즈노트 상품군 | 매출 비중 성격 | 스튜디오 사정권 | 판정 |
|---|---|---|---|
| **맞춤 제휴 광고** (리드수집·체험단·리서치·브랜드위크) — "별도 페이지 구성, CPT 협의" | 고단가·저볼륨 | **정면 사정권** | 🟢 **지금 90% 되어 있다** |
| **패널 내 지면** (키즈노트엔 없는 축 — 우리 고유) | 신규 | **정면 사정권** | 🟢 구조적 우위 |
| **DA 노출형** (스플래시·메인배너·리스트배너·종료팝업·혜택탭) | 볼륨 주력 | 앱 셸 소유 | 🔴 **스튜디오 밖** |

한 줄로: **스튜디오는 "광고 소재·랜딩을 개발자 없이 찍어내는 공장"으로는 이미 거의 완성이고, "광고를 팔고 노출하고 정산하는 시스템"으로는 아직 0이다.** 전자는 이번 분기에 회수 가능하고, 후자는 별도 제품이다.

## 1. 키즈노트 상품의 해부 — 광고는 3층이다

소개서를 지면 목록으로 읽으면 오해한다. 광고 상품은 언제나 세 층이 겹쳐 있고, **층마다 소유자가 다르다.**

```
[3층] 운영·정산   인벤토리 부킹 → D-3 소재 → D-1 검수 → D-Day 라이브 → 리포트 어드민
                  위약금 규정 · 대행수수료 20/30% · 제한업종 · 소재 주 2회 교체
                                                                    ← 광고운영 조직 + 콘솔
[2층] 서빙·측정   타게팅(지역·자녀월령·관심사·알림장키워드·결제이력) · 노출/클릭 카운팅
                  CPM 2,000~50,000원 · CPC 100원 · CPT 1,000만원/일 · CTR 0.12~15%
                                                                    ← 애드서버 + 데이터
[1층] 지면·소재   스플래시 · 메인배너 · 포토엔딩 · 혜택탭 · 랜딩페이지 · 제휴 상세페이지
                                                                    ← 앱 셸 / 저작도구
```

**패널 스튜디오는 1층 도구다.** 그것도 1층 전체가 아니라 "앱 셸이 아닌 지면"에 한한다.

키즈노트 단가표를 이 렌즈로 다시 보면 구조가 선명해진다:

- CPT 최고가 3종(스플래시 1,000만/일, 인트로PKG 1,200만/일, 메인팝업 1,000만/일)은 **전부 앱 실행 직후 전면 지면**이다. 앱 본체가 소유한다.
- CPM 최고가(프리미엄 포토엔딩 50,000원, CTR 15%)는 **키즈노트 고유 행동 맥락**(아이 사진 스와이프 끝)에 붙어 있다. 지면 자체가 서비스 고유성에서 나온다.
- 반면 "맞춤 제휴 광고"는 단가가 **"협의"** 이고 상품 설명이 `별도 페이지 구성`이다. — 여기가 저작도구의 자리다.

## 2. 트래픽 구조가 다르다 — 같은 상품표를 베끼면 안 된다

| | 키즈노트 | 헤이홈 (스마트홈) |
|---|---|---|
| 방문 동기 | 알림장·사진 — **매일 강제 방문**, 일 3.5회 | 기기 제어 — **목적지향·단발**, 자동화될수록 앱을 안 연다 |
| 체류 | 사진 열람 등 몰입 구간 존재 | 짧다. 껐다 켰다 하고 나간다 |
| DAU | 160만 (MAU 260만) | 확인 필요 — **자릿수가 다를 것으로 가정** |
| 1st-party 축 | 자녀 월령·호칭·지역·알림장 키워드·결제이력 | **보유 기기·설치 공간·사용 패턴·소모품 주기·구매이력** |
| 구매력 시그널 | 추정(20~40대 여성 80%) | **검증됨** — 기기를 실제로 산 사람 |

**결론: CPM 볼륨 장사는 불리하고, 전환형·제휴형이 유리하다.** 노출 수로 파는 상품은 DAU가 곧 재고인데 그 재고가 얇다. 대신 헤이홈에는 키즈노트에 없는 것이 있다 — **기기 상태(DP)라는 실시간 구매 의도 신호**. 필터 교체 알림이 뜬 사용자에게 필터를 파는 것은 CTR 15%짜리 포토엔딩보다 전환이 좋을 수 있다.

그리고 자사 커머스(`m.hej.life`)가 있다. **첫 광고주는 외부가 아니라 우리 자신이어야 한다** — 측정 인프라 없이도 매출 귀속이 자체 확인 가능하고, 실패해도 광고주 신뢰를 잃지 않는다.

## 3. 스튜디오가 지금 실제로 가진 것 (코드 확인)

리포지토리를 실사한 결과, 광고 관련 자산은 **문서에 적힌 것보다 많다.**

| 자산 | 위치 | 상태 |
|---|---|---|
| 광고 위젯 4종 `adBanner`·`adHero`·`adPoint`·`ctaButton` | `web/studio.html:964-1010`, `src/emit.mjs:371-378` | ✅ 프리뷰 + 실제 `.tsx` 생성 |
| PROMO 프로젝트 유형 (광고 랜딩 자체가 패널) | `web/studio.html:545-570` | ✅ 히어로+혜택+CTA 구성 |
| **상세페이지 마법사** — 기획자용 5단계 안내형 | 커밋 `71760e4` | ✅ 비개발자 단독 저작 |
| **KB 제품검색 → 히어로·혜택 자동 제안** | 커밋 `2836f57`, `server/index.mjs:359` | ✅ 사내 KB 프록시(토큰 서버측) |
| 광고 에셋 WebP 파이프라인 (P8) | `src/assets.mjs` | ✅ 정지=브라우저 Canvas, 애니 GIF/영상=CLI |
| 링크 모델 (`url`·`desc`·`promo`·`price`·`image`·`mode`) | 저작 모델 `links` | ✅ |
| URL 프레임 프리체크 A/B/C (XFO·CSP) | `src/precheck.mjs` | ✅ 랜딩 임베드 가능여부 사전판정 |
| 백엔드 (저장·구글 OIDC·생성·**실 ray build**) | `server/index.mjs`, 커밋 `101311b` | ✅ SaaS 형태로 이미 운영 가능 |

즉 **"광고주 제품 → 랜딩 상세페이지"** 경로는 이미 뚫려 있다. 마법사에 제품명을 넣으면 KB에서 사양을 끌어와 히어로 카피와 혜택 포인트를 제안하고, 이미지를 올리면 WebP로 굽고, 프리뷰로 확인하고, Ray 저장소나 dist를 받는다. **키즈노트가 "맞춤 제휴 광고 — 키즈노트 전용 소재 기획 지원"이라 부르며 사람이 하는 일을, 우리는 도구로 한다.**

## 4. 결정적 제약 — 다섯 가지, 정직하게

### 제약 1. 빌드타임 정적 바인딩 (가장 크다)

`emitLinks()`가 광고 URL을 **코드 상수로 굽는다**:

```js
// src/emit.mjs:117-127
'export const WEB_LINKS = {',
  `  ${k}: '${v.url}',`,   // ← URL 이 번들에 박힌다
'} as const;',
```

이미지도 마찬가지로 `src/res/ad-<key>.webp` 로 번들에 들어간다(P8).

**따라서 소재 교체 = 재빌드 + Tuya 심사 + 앱 배포.** 키즈노트가 Q&A 첫 항목으로 "소재 교체 주 2회 가능"을 명시한 것과 정면으로 충돌한다. 광고는 소재를 바꾸는 사업이다.

> 지금 구조는 **광고 서빙(ad serving)이 아니라 광고 퍼블리싱(publishing)** 이다. 이 한 줄이 이 평가의 핵심이다.

### 제약 2. 측정이 0이다

코드베이스 전체에서 `impression` / `click` / `tracking` / `CPM` / `CPC` 검색 결과 **0건**.

광고 상품에서 리포트는 부가기능이 아니라 상품의 절반이다. 키즈노트 소개서도 D-Day 항목에 "노출 수, 클릭 수, 클릭률 데이터 확인이 가능한 어드민 계정 별도 전달"을 못박는다. **측정 없이는 CPM·CPC 과금 자체가 성립하지 않는다.** CPT(기간 정액)만 팔 수 있는데, CPT는 DAU가 받쳐줘야 팔린다.

### 제약 3. 타게팅 축이 없다

패널은 **기기 단위**로 렌더된다. 사용자 속성에 따라 다른 소재를 내보낼 분기점이 모델에 없다. 헤이홈에 타게팅 재료(보유 기기·공간·사용 패턴)는 존재하지만, 스튜디오에 그 개념이 아직 없다.

### 제약 4. 운영 워크플로가 통째로 없다

인벤토리 확인 → 부킹 → 소재 검수(자동+수동) → 랜딩 검수 → 라이브 → 사후 모니터링 → 위약금·정산. 키즈노트 소개서 18~21p가 전부 이 이야기다. 스튜디오는 **만드는 도구**지 **파는 시스템**이 아니다.

### 제약 5. 지면 소유권 — 앱 셸은 우리 것이 아닐 수 있다

키즈노트 매출 주력(스플래시·메인배너·종료팝업·혜택탭)은 전부 앱 셸 지면이다. 패널 스튜디오는 **미니앱 패널 내부**만 만진다. 헤이홈 앱 메인/스플래시에 광고 지면을 낼 수 있는지는 앱팀 소관이며, Tuya 기반 셸이라면 제어권이 제한될 수 있다. — **미확인 가정. 앱팀 확인 필요.**

## 5. 그래서 무엇이 타당한가

### 🟢 지금 당장 타당 — 제휴형 랜딩 공장

키즈노트의 "맞춤 제휴 광고"(리드수집·리서치·체험단·브랜드위크·퀴즈)는 전부 **별도 페이지**다. 스튜디오는 이미 이걸 만든다. 부족한 건 광고 특화 템플릿과 검수 체크리스트뿐이다.

가치는 광고 매출 이전에 **내부 비용 절감**에서 먼저 나온다 — 마케팅팀이 개발자 대기 없이 랜딩을 찍어낸다.

### 🟢 구조적 우위 — 패널 내 컨텍스트 지면

키즈노트에 없는 축이다. 이미 레퍼런스 패널이 이 패턴을 쓰고 있다:

```js
// web/studio.html:534-535 — haatz 설정 화면
{ id:'w11', type:'linkTile', link:'service' },   // A/S
{ id:'w12', type:'linkTile', link:'filter'  },   // 소모품 구매
{ id:'w12b', type:'adBanner', link:'shop'   },   // 광고 배너
```

여기에 DP를 물리면 **상태 기반 광고**가 된다 — `filter_reset_notice` DP가 서면 필터 광고, 전력 사용량이 높으면 절전 제품. 키즈노트가 자녀 월령으로 하는 일을 우리는 기기 상태로 한다. **스튜디오가 DP 바인딩을 이미 갖고 있어 구조적으로 유리하다.** 단, 제약 1을 깨야 실현된다.

### 🔴 지금은 타당하지 않음

- **3rd파티 광고주 CPM 판매** — 측정·서빙·리포트 없이는 팔 수 없다. 순서가 틀렸다.
- **스플래시·메인배너·종료팝업·푸시** — 스튜디오 사정권 밖. 앱팀/CRM 과제.

## 6. 실행안

### Phase A — 광고 랜딩 공장 정식화 (2~3주, 현 코드 위)

기존 자산 재조립. 신규 인프라 0.

1. **광고 템플릿 3종 추가** — `panels/templates/` 에 리드수집형·체험단형·혜택/쿠폰형. `npm run p9`가 이미 템플릿 왕복을 게이트한다.
2. **마법사에 광고 모드** — 현 5단계(제품→히어로→혜택→이미지→링크)에 캠페인 기간·광고주·랜딩 URL 필드 추가.
3. **소재 검수 체크리스트 자동생성** — `src/handoff.mjs`의 gap→체크리스트 패턴을 그대로 재사용해 `ADREVIEW.md` 를 낸다. 키즈노트 소재 가이드(20p)에 대응.
4. **프리체크를 검수 게이트로** — 랜딩 URL의 A/B/C 판정(`src/precheck.mjs`)을 광고 검수 필수 항목으로.

**회수:** 마케팅팀 랜딩 제작 리드타임 단축. 광고 매출 없이도 흑자.

### Phase B — 동적 광고 슬롯 (4~6주) ★ 결정적 분기

제약 1·2·3을 **한 번의 구조 변경으로** 동시에 푼다.

```
지금:  WEB_LINKS = { shop: 'https://...' }        // 코드에 박힘, 교체하려면 재빌드
바꿈:  <AdSlot id="panel-setting-bottom" />       // 슬롯 ID 만 굽는다
              │ 런타임
              ▼
       GET /api/ads/serve?slot=&deviceType=&productId=
              ▼
       { creative: {img, title, url}, impressionId }
              │
       POST /api/ads/event  {impressionId, type: view|click}
```

- 빌드타임에 굽는 것은 **슬롯 ID 뿐**. 소재는 런타임에 온다 → **소재 교체에 재심사 불필요.**
- 슬롯 요청에 `deviceType`·DP 상태를 실으면 **컨텍스트 타게팅**이 성립.
- `impressionId` 왕복이 곧 **측정** → CPM/CPC 과금 가능.

**⚠️ 선행 확인(블로커 후보): Tuya 미니앱 심사가 런타임 원격 콘텐츠 로딩을 허용하는가.** 금지라면 Phase B는 성립하지 않고, 대안은 "슬롯에 N개 소재를 미리 굽고 런타임에 로테이션"이라는 반쪽짜리가 된다. **Phase B 착수 전 이것부터 확인할 것.**

**첫 광고주는 우리 자신** — `m.hej.life` 자사 커머스. 측정 신뢰도를 매출 귀속으로 자체 검증한 뒤 외부에 판다.

### Phase C — 광고 운영 콘솔 (분기 단위, 별도 제품)

캠페인 관리·인벤토리 부킹·소재 검수 큐·리포트 어드민·정산. **스튜디오가 아니다.** 스튜디오는 이 제품에 "소재 제작기"로 붙는다. Phase B 실측 데이터로 지면 가치가 증명된 뒤에 착수한다.

## 7. 미해결 가정 (의사결정 전 확인 필요)

1. **헤이홈 앱 MAU/DAU** — 지면 재고의 절대량. CPM 상품 성립 여부가 여기서 갈린다.
2. **앱 셸 지면 제어권** — 스플래시·메인배너를 우리가 낼 수 있나. Tuya 셸이면 제한적일 수 있다.
3. **Tuya 미니앱 심사의 원격 콘텐츠 정책** — Phase B의 전제.
4. **광고 목적 개인정보 활용 동의 범위** — 현 헤이홈 약관이 광고 타게팅을 커버하는가. 키즈노트는 이 조건(수집 고지·1년·즉시 파기)을 19p에 별도 명시했다.
5. **광고 영업 조직 유무** — 도구가 있어도 파는 사람이 없으면 매출은 0이다.

## 8. 권고

**Phase A 즉시 착수.** 비용이 사실상 0이고(이미 90% 구현됨), 광고 사업 성패와 무관하게 내부 랜딩 제작 비용을 회수한다. 되돌릴 것도 없다.

**Phase B는 조건부.** 위 가정 3번(Tuya 심사)을 먼저 확인하고, 통과하면 **자사 커머스를 첫 광고주로** 착수한다. 이 단계가 "패널 스튜디오로 광고를 붙인다"가 진짜로 성립하는지를 가르는 분기다.

**Phase C 및 외부 광고주 CPM 판매는 보류.** DAU 규모와 측정 실측치 없이 상품표를 만드는 것은 순서가 거꾸로다. 키즈노트가 저 단가표를 낼 수 있는 이유는 DAU 160만과 어드민 리포트를 이미 갖고 있기 때문이지, 지면 목록이 길어서가 아니다.

### 한 문장

패널 스튜디오는 **키즈노트 상품표의 "맞춤 제휴 광고" 칸을 도구로 자동화하는 데는 이미 준비되어 있고, 나머지 칸(DA·메시지)은 애초에 다른 팀의 과제다.** 스튜디오를 광고 플랫폼으로 키우는 유일한 관문은 **빌드타임 정적 바인딩을 런타임 슬롯으로 바꾸는 Phase B** 하나이며, 그 전제는 Tuya 심사 정책 확인이다.
