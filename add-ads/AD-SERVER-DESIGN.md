# 광고 서버 설계 — 패널은 문(門)만 갖고, 광고는 서버가 판다

> 2026-07-23 · 임베디드-엣지AI파트
> 선행 문서: [`FEASIBILITY.md`](FEASIBILITY.md) (타당성 검토 — 이 문서가 그 설계 파트를 대체한다)

## 0. 이 설계가 바로잡는 것

앞선 검토서는 **"소재를 바꾸려면 재빌드 → 심사 → 앱 배포"** 라고 썼다. **틀렸다.**

| 초판 서술 | 사실 |
|---|---|
| 패널 변경이 네이티브 앱 배포에 묶인다 | **아니다.** 미니앱 패널은 앱과 독립적으로 갱신된다 |
| 그래서 런타임 슬롯 도입이 "관문"이다 | 관문이 아니었다. **더 나은 답이 따로 있다** |

**단, 반대 방향으로 과하게 정정하지 않는다.** 이 문서 초고는 "자체 Cube 를 소유하니 게시
파이프라인도 우리 것"이라고 썼는데 **그것도 틀렸다.** Cube 는 **런타임 IoT 클라우드**지
미니앱 패널을 발행하는 개발 플랫폼이 아니다. 실측 현황([`docs/deploy-pipeline-status.md`](../docs/deploy-pipeline-status.md)):

| 단계 | 자동화 |
|---|---|
| 저작 → 생성 → 에셋 → **빌드**(`ray build -t tuya`) | ✅ 헤드리스로 된다 |
| **업로드 → PID 바인딩 → 릴리스** | ⛔ **Tuya MiniApp IDE 수동.** `ray` CLI 에 발행 커맨드가 없다 |
| 발행에 Tuya 심사가 개입하는가 | ❓ **미확인** — Tuya 문의 ⓓ 항목 |

**이 사실은 이 설계의 근거를 약화시키지 않는다. 강화한다.** 패널 게시가 아직 사람 손을
타고 심사 개입 여부조차 모르는 상태라면, **소재 교체를 패널 재배포에 묶는 설계는 더더욱
안 된다.**

**그러나 패널 갱신이 싸졌다고 해서 "소재를 패널에 굽는" 설계가 옳아지지는 않는다.**
번들은 모두에게 똑같고 되돌아오는 신호가 없다. 재배포가 아무리 싸도 다음은 그대로 불가능하다:

- 사람마다 다른 소재 (번들은 하나다)
- 노출·클릭 집계 (번들은 보고하지 않는다)
- 빈도 제한 · 캠페인 기간 · A/B · 소진 관리
- 마케터가 **엔지니어 없이** 캠페인을 켜고 끄는 것

배포 주기가 아니라 **배포라는 행위 자체**가 광고 운영과 맞지 않는다. 광고는 분 단위로,
비개발자가, 캠페인 단위로 움직인다. 그래서 결론은 하나다.

> **패널은 바뀌지 않는 문(door)만 갖는다. 문 뒤에서 무엇을 보여줄지는 광고 서버가 정한다.**

## 1. 구조

```
 미니앱 패널 (Ray · 앱과 독립적으로 갱신)
   └ 광고 슬롯 = 고정 링크 하나
        https://ads.hej.life/go?slot=panel.setting.bottom&d=<deviceId>&p=<productId>
                    │  (패널은 이 URL 말고 아무것도 모른다)
                    ▼
 ┌──────────────────────────────────────────────────────────┐
 │  광고 서버 (별도 웹서버 — 여기가 매일 바뀐다)                │
 │                                                          │
 │   /go      결정 → 클릭 기록 → 랜딩으로                     │
 │   /l/<id>  랜딩 페이지 (패널 스튜디오가 만든 것)             │
 │   /e       이벤트 수집 (열람·전환)                         │
 │   /admin   캠페인·소재·타게팅·리포트 (마케터용)             │
 │                                                          │
 │   결정 엔진 ─ 타게팅 ─ 빈도 ─ 기간 ─ 우선순위               │
 │   저장소   ─ 캠페인 · 소재 · 이벤트 · 전환                  │
 └──────────────────────────────────────────────────────────┘
                    │ 신원·속성 조회
                    ▼
        Cube (우리 소유) · 헤이홈 커머스(m.hej.life)
```

**패널에 남는 것은 슬롯 ID 와 도메인뿐이다.** 캠페인이 바뀌어도, 광고주가 바뀌어도,
소재가 바뀌어도 패널은 그대로다. 패널 재배포는 **슬롯의 위치를 옮길 때만** 일어난다.

## 2. 이 설계가 여는 것 — 측정이 오늘 당장 된다

앞선 검토서의 "측정 0 → 팔 수 없다"도 절반은 틀렸다. **클릭은 광고 서버에 도착하는 순간 측정된다.**
패널을 하나도 안 바꿔도 된다.

| 지표 | 지금 설계로 | 비고 |
|---|---|---|
| **클릭** | ✅ 즉시 | `/go` 가 곧 클릭이다 |
| **전환** | ✅ 즉시 | 자사 커머스는 우리 것이라 주문까지 이어 붙인다 |
| **랜딩 체류·스크롤·폼 완료** | ✅ 즉시 | 랜딩이 우리 페이지다 |
| **노출(impression)** | ⚠️ 조건부 | 패널 안에서 그려진 것은 서버가 모른다 (§6) |

**그래서 CPC·CPA 상품은 지금 구조로 팔 수 있다.** 키즈노트가 혜택 탭을 CPC 100원에 파는
바로 그 자리다. CPM 만 노출 신호가 필요하고, 그건 §6 의 선택지다.

## 3. 패널 ↔ 서버 계약

### 3.1 슬롯

슬롯 ID 는 **지면의 이름**이지 캠페인 이름이 아니다. 오래 안 바뀌어야 한다.

| 슬롯 ID | 자리 |
|---|---|
| `panel.<product>.home.banner` | 제품 홈 화면 배너 |
| `panel.<product>.setting.bottom` | 설정 화면 하단 |
| `panel.<product>.consumable` | 소모품 교체 안내 (필터·배터리) |
| `panel.<product>.interstitial` | 전면 프로모션 |

### 3.2 나가는 URL

```
GET https://ads.hej.life/go
      ?slot=panel.airpurifier.setting.bottom   # 필수 — 어느 자리인가
      &d=<deviceId>                            # 선택 — 누구인가(가명)
      &p=<productId>                           # 선택 — 무슨 제품인가
      &s=<dpHint>                              # 선택 — 지금 상태 (예: filter_life=0)
      &v=1                                     # 계약 버전
```

**패널이 보내는 값은 전부 '힌트'다. 신뢰하지 않는다.** 패널은 클라이언트 코드라
누구나 고쳐 부를 수 있다 — 서명 키를 숨길 곳이 없다. 그래서:

- 과금·타게팅의 **권위 있는 속성은 서버가 Cube 에 다시 물어 확인**한다(디바이스→제품→소유자).
- 힌트는 조회를 줄이는 용도, 그리고 **확인에 실패하면 광고를 안 내보내는 근거**로만 쓴다.
- CPC 과금 캠페인은 §7 의 부정클릭 방어를 반드시 통과시킨다.

### 3.3 응답

```
302 → https://ads.hej.life/l/<creativeId>?imp=<impressionId>   (우리 랜딩)
302 → https://m.hej.life/product/...?utm=...&imp=<impressionId>  (자사 커머스)
302 → <광고주 랜딩>?imp=<impressionId>                          (외부 광고주)
204 → 내보낼 광고 없음 (패널은 조용히 실패)
```

`impressionId` 가 이후 모든 이벤트를 하나로 묶는 열쇠다.

## 4. 랜딩은 패널 스튜디오가 만든다

이번에 붙인 **광고 포맷 팩(위젯 7종·템플릿 7종)** 의 목적지가 바뀐다.

```
지금:  스튜디오 → Ray 패널 저장소 → 번들에 구움      (바꾸려면 재배포)
설계:  스튜디오 → 광고 서버에 랜딩 게시              (바꾸면 즉시 반영)
```

포맷은 그대로 쓴다 — 리드수집 · 체험단 · 리서치 · 혜택쿠폰 · 브랜드위크 · 혜택탭 · 전면.
**이것들은 원래 웹페이지였다.** Ray 패널로 굽던 것이 오히려 우회로였다.

- 스튜디오의 저작 모델(`*.studio.json`)을 광고 서버가 **HTML 로 렌더**한다.
  (`emitAdStyles` 가 이미 테마 색 기반 CSS 를 낸다 — 그대로 재사용)
- `adConsent` 위젯이 여기서 진짜 값을 한다. 리드 폼이 실제로 개인정보를 받으므로
  고지·동의가 **법적 요건**이 된다.
- `adStepGuide`·`adFactRow`·`adCoupon` 은 웹에서 그대로 산다.

**덤: 임베드 제약이 사라진다.** 리포의 P2 프리체크(A/B/C)는 3rd파티가 `X-Frame-Options`
로 막는 문제였는데, **랜딩이 우리 서버면 헤더를 우리가 정한다.** `frame-ancestors` 를 열어
`embedded-selfhosted` 로 앱 안에서 열 수 있다 — 외부 브라우저로 튕겨 나가지 않아 이탈이 준다.

## 4-1. 스택 — Go. 단, 렌더는 Node 에 남긴다

**광고 서버는 Go 로 쓴다.** 사내에 이미 Go 가 있고(`teamgoqual/uiot` — `cmd/`+`internal/`, pgx),
핫패스가 전부 데이터 처리라 단일 바이너리·저메모리·동시성이 그대로 이득이다.

**다만 한 가지를 Go 로 옮기지 않는다 — 랜딩 렌더다.**

| 책임 | 스택 | 왜 |
|---|---|---|
| 저작 · **랜딩 HTML 렌더** | **Node** (기존 스튜디오) | 위젯 카탈로그·`emitAdStyles` 가 이미 여기 있다. Go 로 포팅하면 렌더 규격이 **두 벌**이 된다 |
| 결정 · 리다이렉트 · **트래킹** · 서빙 | **Go** | `/go`·`/e` 는 QPS 가 붙는 자리. 템플릿 렌더가 없어 단순하고 빠르다 |

**핵심: 랜딩은 발행 시점에 스튜디오가 HTML 로 구워서 광고 서버에 올린다.** 광고 서버는
`Creative.LandingHTML` 을 그대로 서빙만 한다. 이러면 소재 교체는 여전히 즉시(스튜디오가
재발행)이면서, 위젯 렌더 SSOT 는 한 곳에 남는다.

`emitAdStyles(panel, { unit:'px' })` 로 같은 규격에서 웹용 CSS 도 나온다 — 패널은 `rpx`,
웹은 `px`. 스타일시트를 두 벌 쓰지 않는다.

> 전부 Node 로 가는 선택지도 가능하다(스튜디오 백엔드에 얹기). 1단계 한정으로는 그게 더
> 빠르지만, `/go`·`/e` 는 결국 광고 트래픽 전량을 받는 자리라 처음부터 갈라 두는 편이 낫다.
> 사내 Go 운영 지식이 이미 있다는 점도 같은 방향을 가리킨다.

## 4-2. 지표 트래커

**리포트는 부가기능이 아니라 상품의 절반이다.** 키즈노트도 D-Day 에 "노출 수·클릭 수·
클릭률을 볼 수 있는 어드민 계정"을 광고주에게 준다.

`impID` 하나가 사슬 전체를 묶는다:

```
impression(나중) → click → landing_view → engage → lead / convert
```

| 이벤트 | 언제 | 과금 |
|---|---|---|
| `impression` | 패널에 떴다 — §6 의 B·C 를 붙여야 들어온다 | CPM |
| `click` | `/go` 를 지났다 | **CPC** |
| `landing_view` | 랜딩이 실제로 열렸다 | — (클릭→도달 이탈 측정) |
| `engage` | 스크롤·체류 | — |
| `lead` | 폼 제출 | CPA |
| `convert` | 구매 (금액 동반) | **CPA** |

파생 지표: `CTR = 클릭/노출` · `도달률 = 랜딩조회/클릭` · `전환율 = 전환/클릭`.
**집계는 이벤트에서만 파생한다** — 별도 카운터를 두지 않는다(어긋날 여지를 만들지 않는다).

세 가지 규율:

1. **원본 식별자를 저장하지 않는다.** deviceID·accountID 는 **솔트 HMAC** 으로만 남고,
   솔트는 **날마다 회전**한다 → 날짜를 넘는 개인 추적이 불가능하다. 대신 빈도 제한·중복
   제거도 하루 범위로 제한된다 — 의도한 맞교환이다.
2. **무효 클릭을 버리지 않고 표시한다.** 같은 (기기·슬롯·캠페인) 클릭이 30초 안에 반복되면
   `billable=false` + 사유. **광고주 리포트는 유효분만, 내부 감사는 전량**을 본다.
   기기를 식별할 수 없으면 중복 판정이 불가하므로 보수적으로 무효 처리한다.
3. **클라이언트를 믿지 않는다.** `/e` 로 올라오는 전환은 `impID` 로 사슬을 거슬러 캠페인·
   소재·기기를 서버가 채운다. 사슬에 없는 `impID` 는 404 — 전환을 위조할 수 없다.

## 4-3. 계정 프로파일 — 자리만 먼저 (소스 미정)

**IoT 플랫폼의 진짜 무기가 여기다.** 이 계정이 어떤 제품을 갖고 있고, 그중 무엇을 실제로
**잘 쓰며**, 어떤 **섹터에 관심**이 있는지. 키즈노트가 "자녀 월령"으로 **추정**하는 자리를
우리는 실사용 데이터로 **확정**할 수 있다.

**소스는 아직 정하지 않았다** — Cube 런타임 API 인지, 데이터웨어하우스인지, 별도 집계
배치인지. 그래서 **자리(seam)만** 만들었다:

```go
type Provider interface {
    Profile(ctx, accountHash) (*Profile, error)   // 모르면 ErrNoProfile
    Name() string                                  // 리포트에 소스가 드러난다
}

type Profile struct {
    Categories []string                 // 보유 제품군
    Usage      map[string]UsageLevel    // 제품군 → 사용 강도 (none|light|heavy)
    Sectors    []Sector                 // 관심 섹터 + 점수(0~1)
}
```

구현체는 지금 `Stub`(항상 "모른다")과 `Static`(테스트용)뿐이다. 소스가 정해지면
`CubeProvider`/`DWProvider` 를 꽂고 `main.go` 의 한 줄만 바꾼다.

타게팅 규칙은 이미 이 신호를 받는다 — `ownsCategory` · `usesHeavily` · `sector`+`sectorMin`.

> **규율: 모르면 매칭하지 않는다(fail closed).** 프로파일을 요구하는 타게팅인데 프로파일이
> 없으면 광고를 **내보내지 않고** 사유를 남긴다. 조용히 전체 노출로 새면, 광고주에게는
> "타게팅했다"고 하면서 실제로는 아무나에게 나가는 **사기**가 된다. 리포트에 프로파일
> 소스명(`stub`)이 함께 나가므로 "지금 타게팅이 실제로 도는지"를 숨길 수 없다.

## 5. 데이터 모델 (최소)

```
campaign     id · 광고주 · 기간 · 상태 · 과금방식(CPC/CPA/CPT) · 예산 · 일일한도
creative     id · campaign · 포맷 · 저작모델(JSON) · 랜딩URL · 검수상태
placement    campaign · slot · 우선순위 · 타게팅규칙 · 빈도제한
event        impressionId · type(click|view|convert) · slot · creative
             · deviceHash · productId · ts · (전환금액)
```

- `deviceHash` — 원본 deviceId 를 저장하지 않는다. 솔트를 넣어 해시하고 **회전**시킨다.
- 리포트는 `event` 집계로 나온다. 광고주 화면은 그 위에 얹는다.

## 6. 노출(impression) — 유일하게 남는 선택지

패널 안에서 배너가 그려진 사실은 서버가 모른다. 셋 중 하나를 고른다.

| 안 | 방법 | 대가 |
|---|---|---|
| **A. 안 판다** | CPC·CPA·CPT 만 판매 | 가장 단순. **1단계 권장** |
| **B. 패널이 알린다** | 슬롯 렌더 시 `/e?type=view` 1회 호출 | 패널이 네트워크를 탄다 — 오프라인 대비·지연·실패 무시 필요 |
| **C. 슬롯을 서버가 그린다** | 패널이 슬롯 내용을 서버에서 받아 렌더 | 완전한 동적 지면 + 노출 측정. 대신 패널 렌더가 서버에 의존 |

**1단계는 A 를 권한다.** 패널을 건드리지 않고 광고 사업을 시작할 수 있고, CPC 는 실제로
팔리는 상품이다. B·C 는 지면 가치가 데이터로 증명된 뒤에 붙인다.

C 를 하더라도 **패널은 오프라인에서 동작해야 한다**(QA 체크리스트가 이미 요구한다).
슬롯은 실패 시 **조용히 사라지는** 것이 기본값이어야 하며, 기기 제어를 절대 막지 않는다.

## 7. 지켜야 할 것

**개인정보.** 보유 기기 기반 타게팅은 개인정보 처리다. 광고 목적 활용 동의가
현재 약관에 있는지 **확인이 선행**이다(검토서 미해결 항목 3). 없으면 동의를 받거나,
동의 전까지는 **비타게팅**(전체 동일 소재)으로만 운영한다 — 그래도 CPC 는 성립한다.

**부정 클릭.** CPC 과금은 클릭을 위조하면 돈이 샌다. `/go` 에 디바이스 확인·빈도 제한·
중복 제거(같은 `deviceHash`+`slot` 단시간 반복 무효)를 넣고, 과금은 **검증 통과분만** 센다.

**소재 검수.** 제한 업종(사금융·성인·사행성·주류)과 소재 가이드는 광고 서버의 검수 상태로
관리한다. 스튜디오가 내는 검수 체크리스트를 그 입력으로 쓴다.

**장애.** 광고 서버가 죽어도 **기기 제어에는 영향이 없어야 한다.** 슬롯은 링크일 뿐이고,
최악의 경우 링크가 안 열리는 것으로 끝나야 한다. 안 A 를 고르면 이 성질이 공짜로 따라온다.

## 8. 단계

| 단계 | 내용 | 패널 변경 |
|---|---|---|
| **1. 링크 + 랜딩** | `/go` · `/l/<id>` · 클릭·전환 기록. 스튜디오가 랜딩 게시. 첫 광고주 = 자사 커머스 | 슬롯 링크 1회만 |
| **2. 결정 엔진** | 캠페인·타게팅·빈도·기간·우선순위. CPC 과금 성립 | 없음 |
| **3. 운영 콘솔** | 마케터가 직접 캠페인 관리 · 광고주 리포트 | 없음 |
| **4. 노출 측정** | §6 의 B 또는 C. CPM 상품 개시 | 있음 (그때 판단) |

1단계가 끝나면 **패널을 다시 건드릴 일이 사실상 없다.** 그게 이 설계의 요점이다.

   §6 의 B·C(패널이 노출을 알리는 안)로 갈 때 비로소 게시 빈도가 문제가 된다.

> 밖에 있고, 1단계는 패널을 사실상 한 번만 건드린다. 패널 배포 파이프라인이 언제 뚫리든
